# システム回路の実行

## 固定繰り返しレートの実行

ほとんどのIBM Quantumシステムは一定の速度で回路を実行します。 この_繰り返し率_はシステムによって異なりますが、基礎となる実行モデルは同じであり、ここで説明します。 まず、3つの回路を別々のジョブを使用してシステムに送ることを考えます。 以下の例は、さまざまな長さのジョブで何が起こるかを示しています。 一定の繰り返し率のため。 システムの繰り返しレートによって与えられた期間全体が一致するようにするために、回路の開始前に発生する_アイドル時間_の可変量があります。

![../../../\_images/fixed_single_circuit1.png](/images/migration/fixed_single_circuit1.png)

同じ回路が単一のジョブにバッチ処理されると、状況は多少変わります。 この場合、ジョブに含まれる回路は、要求される各ショットの回路を繰り返すことによって実行されます。 実行は、回路とショットの行列に対して列単位で行われます(下記参照)。

![../../../\_images/circuits_shots_matrix1.png](/images/migration/circuits_shots_matrix1.png)

ジョブ内の4つの回路の行列で、回路の実行パターンを示します。

システムに回路のバッチを送信する場合、別々に実行する場合とは異なる方法で実行されます。 すなわち、回路の初期化、すなわちグランド状態の準備、および測定(もしあれば)はすべての回路上で整列されます。

![../../../\_images/fixed_batch_circuit1.png](/images/migration/fixed_batch_circuit1.png)

したがって、各回路はバッチ内の長さと長さが等しくなります。 繰り返しの速度に合うようにすべての回路の前に置かれるアイドル時間があります

## 動的繰り返しレートの実行

IBM Quantum システムによっては、動的繰り返しレートの実行が可能になります。 これらのシステムは Qiskit で`backend.configuration().dynamic_reprate_enabled` を使用して識別され、`True` の値を返します。 これらのシステムでは、送信されたジョブの `rep_delay` を設定することで、上記のアイドル時間を手動で設定することができます。 上記の図から、アイドル時間を短縮することによって、動的繰り返しレートをサポートするシステム上の回路のスループットが向上する可能性があることがわかります。 詳細な使用例については、条件付きリセットの次のセクションを参照してください。
